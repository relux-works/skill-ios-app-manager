### 26189ed Fix AI loaders: prevent freeze, staggered timing, smoother shimmer

**What**: Improves the AI content reveal UX introduced in batch 06 (commit ea4d4a4). Three changes:

1. **Staggered reveal timing**: Replaces single `isAIContentReady` flag with three separate flags: `isMessageReady` (2s), `isOperatorRequestReady` (3s), `isSuggestionsReady` (4s). Each reveals independently with `.easeInOut(duration: 0.3)` animation. Creates a cascading "AI thinking" effect — message drafting appears first, then operator request, then suggestions.

2. **Prevent freeze**: Replaces `Task.sleep` + `@MainActor` pattern with `DispatchQueue.main.asyncAfter`. Removes the `aiRevealTask` lifecycle management (no more cancel/reassign on deinit). The old approach could potentially freeze if the Task was blocked; GCD dispatch avoids this.

3. **Smoother shimmer**: Changes gradient from 3-stop (`clear, white(0.6), clear`) to 6-stop (`clear, clear, white(0.4), white(0.6), white(0.4), clear, clear`) — creates a softer, more gaussian shimmer peak. Widens shimmer frame (2x → 3x width) and travel distance (3x → 4x). Changes animation from `.linear(1.5s)` to `.easeInOut(2.0s)` — smoother acceleration/deceleration.

**Scaffolding assessment**:
- [x] INSTRUCTION

**INSTRUCTION**:
- No ios-app-manager commands — UX polish for AI loading states in CRM module
- Manual edits:
  - `ContactDetailViewModel.swift` — 3 separate ready flags, `DispatchQueue.main.asyncAfter` staggered timers
  - `ContactDetailView.swift` — 3 references updated (`isAIContentReady` → `isMessageReady` / `isOperatorRequestReady` / `isSuggestionsReady`)
  - `View+Shimmer.swift` — 6-stop gradient, wider frame, easeInOut animation
- Notes: The switch from `Task.sleep` to `DispatchQueue.main.asyncAfter` is a deliberate trade-off — loses structured concurrency benefits (cancellation) but gains simplicity and avoids potential UI thread starvation. For fire-and-forget UX delays, GCD is fine. The staggered timing (2s, 3s, 4s) reinforces the "AI thinking" illusion from batch 06.

### 7791e01 XFL-50 Check for active onboarding session before starting new flow

**What**: Adds pre-check for active onboarding sessions before navigating to onboarding flow. Prevents starting a duplicate onboarding when one is already in progress for the same organization.

1. **OnboardingService**: `getStatus()` and `getTelegramStatus()` now accept optional `organizationId` parameter. Passes it as `URLQueryItem` to the status endpoint, allowing server to return organization-specific onboarding status.

2. **OrganizationDetailView**: New `connectMessenger(_:)` method with `isCheckingSession` guard. Before navigating to onboarding, fetches current status for the specific organization. If a non-terminal session exists, navigates with `initialStatus` pre-loaded (resumes existing flow). If no active session, navigates normally (starts new flow).

3. **ProfileView**: Same `connectMessenger(_:organizationId:)` pattern, applied to organization cards. Both views now check for active sessions before starting onboarding.

**Scaffolding assessment**:
- [x] INSTRUCTION

**INSTRUCTION**:
- No ios-app-manager commands — business logic for onboarding session management
- Manual edits:
  - `OnboardingService.swift` — `organizationId` parameter on `getStatus()` and `getTelegramStatus()`
  - `OrganizationDetailView.swift` — `connectMessenger(_:)` with session check (28 lines)
  - `ProfileView.swift` — `connectMessenger(_:organizationId:)` with session check (31 lines)
- Notes: The duplicated `connectMessenger` logic between `OrganizationDetailView` and `ProfileView` (~30 lines each, nearly identical) is a code smell. In a well-structured app, this would be a single method on a shared coordinator or navigation service. The `isCheckingSession` guard prevents double-taps during the async check.
